## Types of Microcopy



### Buttons and CTAs

Buttons are action triggers that tell users what will happen when clicked. Effective button copy is specific, action-oriented, and creates appropriate expectations.

**Button Writing Guidelines:**
- Start with a verb (action word)
- Be specific about the action
- Match button text to user expectations
- Consider button context and placement
- Keep it short (1-4 words ideal)

**Examples:**
| Generic | Specific |
|---------|----------|
| Submit | Send message |
| OK | Save changes |
| Yes | Delete account |
| Click here | Download report |
| Continue | Go to checkout |



### Labels and Form Fields

Labels identify form fields and tell users what information to enter. Good labels are clear, concise, and positioned for easy scanning.

**Label Writing Guidelines:**
- Use sentence case for readability
- Be specific about required format
- Position labels consistently (top-aligned often works best)
- Indicate required vs. optional fields
- Avoid ending labels with colons in modern interfaces

**Examples:**
- "Email address" (not "Email:")
- "Phone number (optional)"
- "Password (8+ characters)"
- "Card number (Visa, Mastercard, or Amex)"



### Placeholder Text

Placeholder text appears inside form fields before users enter information. It should hint at the expected format without serving as the primary label.

**Placeholder Best Practices:**
- Use placeholders to show format examples
- Never use placeholders as the only label
- Keep placeholder text brief
- Use a lighter color to distinguish from user input
- Avoid critical information in placeholders (they disappear on input)

**Examples:**
- Email field: "name@example.com"
- Phone field: "(555) 555-5555"
- Date field: "MM/DD/YYYY"
- Search field: "Search products..."



### Error Messages

Error messages inform users when something goes wrong and guide them toward resolution. Poorly written error messages frustrate users and can cause abandonment; well-crafted ones turn potential friction into moments of support.

**Error Message Framework:**
1. **Explain what happened** (briefly)
2. **Why it happened** (if helpful)
3. **How to fix it** (specific action)

**Before (Unhelpful):**
> "Error 404: Resource not found"

**After (Helpful):**
> "We couldn't find that page. It may have been moved or deleted. Try searching or go back to the homepage."

**Error Message Guidelines:**
- Use plain language, not error codes
- Be specific about the problem
- Provide actionable next steps
- Avoid blaming the user
- Match tone to severity (serious errors warrant serious tone)
- Position errors near the relevant field

**Examples by Type:**

**Validation Error:**
> "Enter a valid email address (e.g., name@example.com)"

**System Error:**
> "Something went wrong on our end. Please try again, or contact support if the problem continues."

**Connection Error:**
> "You seem to be offline. Check your connection and try again."



### Empty States

Empty states appear when there's no content to display—such as an empty inbox, search with no results, or a new account with no data. These moments are opportunities to guide and encourage users.

**Empty State Components:**
1. **Explain the state** - What is this area and why is it empty?
2. **Guide next action** - What should the user do?
3. **Encourage** - Make users feel this is normal and fixable

**Examples:**

**Empty Inbox:**
> "No messages yet
> When you receive messages, they'll appear here. Start a conversation to get going."
> [New message] button

**No Search Results:**
> "No results for 'xyz'
> Try different keywords or check your spelling."

**New Account:**
> "Welcome to your dashboard
> You haven't created any projects yet. Start your first project to see your progress here."
> [Create project] button



### Tooltips and Help Text

Tooltips provide contextual help without cluttering the interface. They appear on hover or tap and should offer quick, useful information.

**Tooltip Guidelines:**
- Keep tooltips brief (1-2 sentences)
- Provide information not obvious from the UI
- Use tooltips sparingly—too many overwhelm users
- Ensure tooltips are accessible (keyboard navigable)
- Position tooltips so they don't obscure relevant content

**Examples:**
- Password field tooltip: "Use at least 8 characters with a mix of letters and numbers."
- Privacy setting tooltip: "When enabled, only people you follow can message you."
- Feature icon tooltip: "Keyboard shortcuts: Press ? to see all shortcuts."



### Confirmation Messages

Confirmation messages reassure users that their action was successful. They close the loop on user intentions and provide peace of mind.

**Confirmation Message Guidelines:**
- Confirm the specific action completed
- Include relevant details (what was saved, sent, deleted)
- Provide next steps if applicable
- Offer undo option for significant actions
- Use positive, encouraging language

**Examples:**
- "Your changes have been saved."
- "Message sent to 3 recipients."
- "Account deleted. We're sorry to see you go."
- "Order confirmed! Check your email for receipt. [Track order]"



### Loading States

Loading states communicate that something is happening and help users wait patiently. Without loading indicators, users may think the interface is broken.

**Loading State Guidelines:**
- Indicate that loading is happening
- Estimate duration when possible
- Explain what's loading (especially for longer waits)
- Consider progress bars for predictable durations
- Add personality when appropriate

**Examples:**
- "Loading..." (basic)
- "Finding the best results for you..."
- "Uploading 3 of 10 photos..."
- "Setting up your account (this takes about 30 seconds)..."
- "Almost there! Just a few more seconds..."



### Onboarding Copy

Onboarding copy welcomes new users and guides them through initial setup. It sets the tone for the relationship and helps users understand core value quickly.

**Onboarding Copy Guidelines:**
- Welcome users warmly
- Explain key features concisely
- Break setup into manageable steps
- Show progress through multi-step flows
- Offer skip options for non-essential steps
- Celebrate completion

**Onboarding Flow Example:**

**Step 1 - Welcome:**
> "Welcome to [Product]!
> Let's get you set up in just a few steps."

**Step 2 - Profile:**
> "Tell us about yourself
> This helps us personalize your experience."

**Step 3 - Preferences:**
> "What are you hoping to accomplish?
> Select all that apply—you can change these anytime."

**Step 4 - Completion:**
> "You're all set!
> Your dashboard is ready. Here are a few things you can do first:"
